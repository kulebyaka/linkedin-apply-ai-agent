# Feature Specification: Auto-Refining Job Filter Prompt

## Overview
- **Feature**: Auto-Refining Filter Prompt + Persistent Notification Center
- **Status**: Draft
- **Created**: 2026-06-08
- **Author**: User + Claude Code

## Problem Statement

The per-user LLM job filter (`UserFilterPreferences.custom_prompt`) is configured once and never learns. Two recurring signals reveal where the filter is wrong, but the system currently discards both:

1. **False positives** — jobs that pass the filter (high score) but the user *declines* at HITL review. The decline currently only flips `status → DECLINED`; no reason is captured and nothing is fed back into the filter.
2. **False negatives** — jobs the filter marks `filtered_out` that the user overrides via "Proceed Anyway" (commit `c5fe155`). The override is a bare button with no captured reason, so the mistake is invisible to the filter.

We want a feedback loop: periodically inspect these signals and **propose** an improved filter prompt that the user reviews and accepts. Because the proposal is generated by a background scheduler job (no active page session), surfacing it requires a notification that survives navigation and reloads — the current toast system (client-only, 5s auto-dismiss, no global store) cannot do this. So this feature has two coupled parts:

- **Part A**: A cross-app, backend-persisted, mark-as-read notification center.
- **Part B**: The auto-refinement scheduler + signal capture + proposal review flow that uses Part A to alert the user.

## Goals & Success Criteria

- Capture *why* a job was declined at HITL (optional reason) and *why* a filtered-out job was a mistake (optional reason on Proceed Anyway).
- Periodically (default weekly) analyze accumulated signals per user and generate a proposed update to the filter prompt's auto-learned section.
- Never silently change a user's hand-written filter prompt — every change is a reviewable proposal accepted explicitly by the user.
- Deliver the proposal via a notification that is visible on any page and persists until the user marks it read / acts on it.
- Gate the whole behavior behind a per-user Settings toggle (default OFF).
- **Success Metrics**:
  - Reduction in HITL decline rate for filter-passing jobs over time for opted-in users.
  - Reduction in Proceed Anyway overrides over time for opted-in users.
  - Proposals are actionable: users accept rather than dismiss the majority of proposals.

## User Stories

1. As a job seeker, when I decline a job at review, I want to optionally note why, so the filter stops surfacing similar jobs.
2. As a job seeker, when I "Proceed Anyway" on a wrongly-filtered job, I want to optionally explain the filter's mistake, so it stops rejecting similar jobs.
3. As a job seeker, I want the system to periodically propose filter improvements based on my decisions, which I can accept or reject — without it ever overwriting my hand-tuned prompt.
4. As a job seeker, I want to opt into (or out of) auto-refinement from Settings.
5. As any user, I want important notifications (like a pending filter proposal) to follow me across pages and stay until I acknowledge them, rather than vanishing after 5 seconds.

## Functional Requirements

### Core Capabilities

**Part A — Persistent Notification Center**
- Backend-persisted, user-scoped notifications stored in a new DB table.
- Two notification tiers that **coexist**:
  - **Transient toasts** (existing `ToastNotification.svelte`): ephemeral confirmations ("Saved", "Copied"). Unchanged behavior; not persisted.
  - **Persistent notifications** (new): acknowledgment-worthy events (filter proposal ready, future: job applied, errors). Stored, mark-as-read, visible app-wide.
- A global notification UI (bell + dropdown/panel) mounted in the root layout, showing unread count and a list with mark-as-read and mark-all-read.
- Notifications carry an optional action link (e.g. proposal → Settings filter section).

**Part B — Auto-Refinement**
- Per-user Settings toggle `auto_refine_enabled` (default `false`).
- Optional reason capture on HITL **decline**, persisted on the job record.
- Optional reason capture on **Proceed Anyway**, persisted on the job record.
- A periodic per-user scheduler job that:
  - Skips users with `auto_refine_enabled == false`.
  - Collects un-consumed decline-with-reason and proceed-with-reason signals since the last refinement.
  - Skips users below the minimum-signal gate (default 10).
  - Feeds at most the most-recent N signals (default cap 50) to the LLM.
  - Generates a proposed update to the **auto-learned criteria block** of `custom_prompt` plus a plain-language rationale.
  - Creates a single pending proposal (supersedes any prior unacknowledged one) and emits a persistent notification.
- A review surface (in the existing `FilterPreferencesSection`) showing the pending proposal: the proposed learned-criteria block, a rationale citing the signals, and **Accept / Reject** actions.
- On **Accept**: the refiner replaces only the delimited auto-learned block inside `custom_prompt`; the user's hand-written portion is untouched. Consumed signals are marked so they aren't reused.
- On **Reject**: proposal discarded; consumed signals marked so they aren't reused.

### User Flows

**Flow 1 — Decline with reason (false positive capture)**
1. User declines a job in HITL review.
2. UI shows an optional reason field (does not block decline).
3. `POST /api/hitl/{job_id}/decide` with `decision="declined"` and optional `reasoning`.
4. Backend sets `status=DECLINED` and persists `decline_reason` + `refine_signal_state="pending"` on the job record (only when a reason is present *or* the job passed the filter — see Data Model).

**Flow 2 — Proceed Anyway with reason (false negative capture)**
1. User clicks "Proceed Anyway" on a `filtered_out` job.
2. The existing modal gains an optional reason text field ("Why was this filtered out by mistake?").
3. `POST /api/jobs/{job_id}/proceed` with optional `override_reason` in the body.
4. Backend persists `override_reason` + `refine_signal_state="pending"` on the job record, then proceeds as today (`skip_filter=True`).

**Flow 3 — Periodic refinement**
1. Scheduler fires (default weekly).
2. For each user with `auto_refine_enabled`, gather pending signals since last refinement.
3. If count `< min_signals` (default 10): skip, update `last_refine_check_at`.
4. Else: take most-recent ≤ `cap` (default 50) signals, call the refiner LLM.
5. Parse proposed auto-learned block + rationale; store as the single pending proposal; mark these signals `refine_signal_state="proposed"`.
6. Emit a persistent notification: "Filter improvement suggested — review in Settings."

**Flow 4 — Review & accept/reject**
1. User opens the notification (or Settings → Filter Preferences).
2. The Filter Refinement panel shows: current learned block, proposed learned block, rationale.
3. **Accept** → `custom_prompt` is rewritten with the new learned block; proposal cleared; signals → `consumed`; success toast.
4. **Reject** → proposal cleared; signals → `consumed`; the linked notification is marked read.

### Data Model

**New: `UserFilterPreferences` field** (`src/models/job_filter.py`)
```python
class UserFilterPreferences(BaseModel):
    natural_language_prefs: str = ""
    custom_prompt: str | None = None
    reject_threshold: int = 30
    warning_threshold: int = 70
    enabled: bool = True
    auto_refine_enabled: bool = False        # NEW — per-user feature flag (default OFF)
```

**Auto-learned block convention** (inside `custom_prompt`)
Delimited, refiner-owned region appended to the user's prompt:
```
<user's hand-written criteria>

<!-- BEGIN auto-learned -->
## Auto-learned criteria
- <refiner-managed bullet, derived from declines/overrides>
<!-- END auto-learned -->
```
The refiner reads/replaces only the content between the markers. If markers are absent, it appends the block. Hand-written text outside the markers is never modified.

**New: `RefinementProposal`** (`src/models/job_filter.py`) — single pending per user, stored as JSON on the user record (e.g. `filter_preferences.pending_proposal` or a sibling JSON column `pending_refinement`)
```python
class RefinementProposal(BaseModel):
    proposed_learned_block: str          # new "## Auto-learned criteria" block
    rationale: str                       # plain-language explanation citing signals
    signal_job_ids: list[str]            # jobs whose signals fed this proposal
    decline_count: int
    override_count: int
    created_at: datetime
```

**HITLDecision** (`src/models/unified.py`) — already has optional `reasoning: str | None`; reuse it as the decline reason. No model change; persistence is added in `_handle_decline`.

**Proceed request body** (`src/api/routes/jobs.py`) — new optional body
```python
class ProceedRequest(BaseModel):
    override_reason: str | None = None
```

**Job record / `Job` table** (`src/services/db/tables.py`) — new columns (added via runtime migration in `src/services/db/migrations.py`, following the `add_user_role` pattern):
- `decline_reason TEXT NULL`
- `override_reason TEXT NULL`
- `refine_signal_state VARCHAR(20) NULL` — one of `pending` / `proposed` / `consumed`; lifecycle marker so a signal is fed to the refiner exactly once.

`JobRecord` (`src/models/unified.py`) gains matching optional fields: `decline_reason`, `override_reason`, `refine_signal_state`.

**New: `NotificationTable`** (`src/services/db/tables.py`)
```python
class NotificationTable(Table, tablename="notification"):
```

## Technical Design

### Architecture

Follows the existing layering: thin FastAPI routes → domain services → repositories (Piccolo). The refiner is a domain method on `JobFilter` invoked by a scheduler job — not a new LangGraph workflow (the work is a single LLM call with structured output, no multi-step state). Notifications are a standard CRUD slice. Signal lifecycle (`pending → proposed → consumed`) lives as a column on the job record, avoiding a separate signals table.

**Refinement cycle (pseudo):**
```
for user in users where auto_refine_enabled:
    signals = repo.list_refine_signals(user.id, state="pending", limit=CAP)   # declines + overrides, newest first
    if len(signals) < MIN_SIGNALS:
        continue
    proposal = job_filter.generate_refinement(signals, user.filter_preferences, user.id)
    user_repo.set_pending_proposal(user.id, proposal)
    repo.mark_signals(proposal.signal_job_ids, state="proposed")
    notifications.create(user.id, type="filter_refinement",
                         title="Filter improvement suggested",
                         action_url="/settings#filter")
```

### Technology Stack
- **Frameworks**: FastAPI (routes), APScheduler (periodic job), Svelte 5 (UI runes stores).
- **Libraries**: Piccolo ORM (tables + migrations), Pydantic v2 (models), existing LLM provider layer.
- **Tools**: existing `PromptLoader` for the new refinement meta-prompt template.

### Data Persistence
- SQLite via Piccolo. New `notification` table + new columns on `job` table, both created idempotently through `src/services/db/migrations.py` (the `_add_column` / `Migration` pattern, recorded in `schema_migrations`).
- Pending proposal and `auto_refine_enabled` stored within the user's `filter_preferences` JSON (no new user column needed) — or a dedicated `pending_refinement` JSON column if cleaner; decide at implementation time.
- Read/unread status persisted server-side so it is cross-device and cross-tab.

## Non-Functional Requirements

- **Performance**: Refinement is a single LLM call per qualifying user, run off the request path on the scheduler. Signal cap (50) bounds prompt size and cost. Min-signal gate (10) prevents noisy/expensive runs on thin data.
- **Security**: All notification and refinement endpoints are user-scoped behind `CurrentUser`; proposals and signals never cross users. Manual admin refine endpoint behind `get_admin_user`.
- **Observability**: Log each refinement cycle per user — signals considered, gate decision, whether a proposal was created, LLM provider used. Log accept/reject outcomes.
- **Error Handling**:
  - LLM failure during refinement: log, leave signals `pending` (retry next cycle), no proposal, no notification.
  - Malformed learned block from LLM: validate markers/structure before storing; reject and log on failure.
  - Accept when `custom_prompt` markers are missing: append a fresh learned block rather than failing.
  - Notification creation failure must not abort the refinement transaction (best-effort, logged).

## Implementation Considerations

### Design Trade-offs

- **Propose-for-review over auto-apply**: avoids silently degrading a hand-tuned filter; the user stays in control. Cost: extra proposal storage + review UI. Chosen for trust and safety.
- **Managed learned-criteria block over full-prompt regeneration**: preserves the user's manual edits verbatim; the refiner only owns a delimited region. Cost: marker bookkeeping. Chosen to never clobber user work.
- **Backend-persisted notifications over client localStorage**: required because the producer is a server-side scheduler with no browser session, and the user expects cross-device/cross-tab persistence. Cost: new table + routes + repository.
- **Two-tier notifications over replacing all toasts**: keeps ephemeral confirmations lightweight and avoids a noisy mark-as-read list and an 8-page migration. Cost: two systems to maintain.
- **Signal state on the job record over a separate signals table**: simpler; each decline/override is already a job, and `pending/proposed/consumed` cleanly models once-only consumption.
- **Single pending proposal over full archive**: simplest model; a new cycle supersedes an unacknowledged proposal. Cost: no audit trail / undo (acceptable for v1).

### Dependencies
- Existing LLM provider layer and `PromptLoader`.
- Existing Piccolo migration framework (`schema_migrations`).
- Existing per-user scheduler iteration in `scheduler.py`.

### Testing Strategy
- **Unit**: `JobFilter.generate_refinement` with mocked LLM (deterministic proposal); learned-block insert/replace logic (markers present, absent, malformed); min-signal gate and cap; signal-state transitions.
- **Unit**: `NotificationRepository` CRUD + mark-read; user-scoping.
- **Integration**: decline-with-reason and proceed-with-reason persist correct fields and signal state; accept rewrites only the learned block; reject clears proposal; both mark signals `consumed`.
- **Integration**: scheduler refinement cycle end-to-end with mocked LLM — gate skip, proposal creation, notification emission.
- **API**: notification endpoints (list/unread-count/read/read-all) and refinement accept/reject with TestClient; auth scoping (401/403).
- **E2E (Playwright)**: notification bell shows pending proposal across navigation and survives reload until acted on; accept flow updates the filter prompt; Proceed Anyway modal reason field; decline reason field.

## Out of Scope
- Full proposal history / audit trail / undo (single pending proposal only).
- Replacing existing transient toasts with the persistent center.
- Learning from *applied* jobs that turned out poorly (only decline + override signals).
- Email/push delivery of notifications (in-app only).
- Auto-tuning numeric thresholds (`reject_threshold` / `warning_threshold`) — refinement targets prompt text only.
- Real-time push (WebSocket/SSE) for notifications — polling on mount + interval is sufficient for v1.
- Cross-user / fleet-wide learning.

## Open Questions
- Notification polling cadence on the frontend (e.g. on mount + every 60s) vs. on-navigation only — pick during implementation; SSE explicitly out of scope.
- Whether to store the pending proposal inside `filter_preferences` JSON or as a dedicated `pending_refinement` column — minor, decide at implementation.
- Should a decline with **no** reason still count as a (weak) signal, or only declines with a reason? Current spec: only declines with a reason (and overrides with a reason) become signals, to keep proposals grounded. Revisit if signal volume is too low to ever hit the min-signal gate.
